# Webnote 开发规范

## 1. 目标

`webnote` 是一个仅部署在 `Cloudflare Workers` 上的云端笔记应用。

核心目标：

- 任意合法路径即笔记地址。
- 主页面在超慢网速下也必须优先加载出文字内容。
- 前台资源极简，后台管理页不做同等级别限制。
- 支持 Markdown 编辑、预览、锁定、自动保存、管理后台、导入导出。
- 全站不依赖传统服务器，只使用 `Workers + Static Assets + D1 + KV + Web Crypto`。

## 2. 已确认的产品规则

### 2.1 路径规则

- 允许任意合法路径作为笔记地址。
- 仅允许字母、数字。
- 保留路径：`admin`、`api`、`static`。
- 长度受 `PATH_MIN_LENGTH` 和 `PATH_MAX_LENGTH` 控制。
- 访问 `/` 时：
  - 优先跳转“最近空白笔记”。
  - 若不存在空白笔记，则生成随机新路径。

### 2.2 空白笔记规则

- 空白定义：去掉空格、换行、制表符后没有文字。
- 空白内容禁止保存。
- “最近空白笔记”按 `created_at DESC` 判定。
- 系统允许创建“空白占位笔记记录”，用于根路径跳转，但不保存正文内容。

### 2.3 编辑与保存

- 输入后 `2 秒` 防抖自动保存。
- 保留手动保存按钮。
- 支持 `Ctrl/Cmd + S` 快捷保存。
- 页面显示：
  - 当前保存状态
  - 最后保存时间
  - 浏览次数

### 2.4 锁定模式

- 访问锁定：必须输入密码才能查看和编辑。
- 编辑锁定：可以查看，但必须输入密码才能编辑。
- 前台支持：
  - 设置锁定
  - 输入密码解锁
  - 解除锁定
  - 锁定模式切换
  - 锁定提示条
  - 解锁与锁定相关模态框

### 2.5 锁定说明

- `hint_text` 在未解锁状态下不可见。
- 未授权用户仅看到通用锁定提示，不看到额外说明。

### 2.6 管理后台

- `/admin`：登录页
- `/admin/dashboard`：后台页
- 单管理员模式
- 后台能力：
  - 获取全部笔记列表
  - 删除笔记
  - 直接修改笔记内容
  - 本地搜索
  - 统计卡片
  - 导入导出
  - 日志查看
  - 退出登录

### 2.7 搜索与导入导出

- 后台搜索范围仅限：
  - 路径
  - 正文纯文本
- 导入导出主格式：`JSON`
- README 最终目标：生产级初始化流程，不是演示级快速跑通说明。

### 2.8 浏览统计与缓存

- 浏览笔记时累计 `view_count`。
- 热点缓存默认策略：
  - `view_count >= 10` 进入 KV 热点缓存
  - `10-49` 次：TTL `5 分钟`
  - `50-199` 次：TTL `30 分钟`
  - `>= 200` 次：TTL `2 小时`
- 保存、加锁、解锁、解除锁定、后台改内容、删除时，必须清缓存。
- 浏览统计口径：
  - 排除管理员访问
  - 同一会话对同一路径 `60 秒` 内重复刷新不重复计数

## 3. 技术方案

### 3.1 平台组成

- `Cloudflare Workers Static Assets`
  - 承载前台页面、后台页面与静态资源
  - 优先用于极轻量首屏加载
- `Cloudflare Worker`
  - 承载所有 API、鉴权、路由决策、保存与锁定逻辑
- `Cloudflare D1`
  - 唯一真相源
  - 存储笔记、锁定信息、管理员会话、后台日志
- `Cloudflare KV`
  - 全局热点缓存
  - 缓存热点笔记的可读视图与元信息快照
- `Web Crypto`
  - JWT 签名与校验
  - 密码哈希、盐值处理、恒时比较

### 3.2 缓存边界

- `D1` 是唯一真相源。
- `KV` 是全局热点缓存。
- `Cache API` 只作为机房本地二级缓存，不承担全局一致性。
- 遇到锁定状态、正文内容、管理后台直接修改等敏感变更，必须优先以 `D1` 为准。

### 3.3 前端技术原则

- 不使用重型前端框架。
- 使用原生 `TypeScript + 模块化 DOM`。
- Markdown 渲染和编辑增强依赖必须极少且可替换。
- 前台与后台资源分离构建，避免后台脚本拖累前台首屏。

## 4. 数据设计

### 4.1 主要表

#### `notes`

- `id`
- `path`
- `raw_content`
- `sanitized_content`
- `rendered_html`
- `is_blank`
- `view_count`
- `created_at`
- `updated_at`
- `last_saved_at`
- `last_viewed_at`

#### `note_locks`

- `note_id`
- `lock_mode`
- `password_hash`
- `salt`
- `hint_text`
- `updated_at`

#### `admin_sessions`

- `session_id`
- `jwt_jti`
- `created_at`
- `expires_at`
- `revoked_at`
- `ip_hash`

#### `admin_logs`

- `id`
- `action`
- `target_path`
- `detail_json`
- `created_at`

### 4.2 安全约束

- 绝不存储明文密码。
- 绝不在源码里写管理员密码或 JWT 密钥。
- 敏感项必须走 `Workers Secrets`。
- 所有 SQL 查询使用参数绑定。
- 所有 HTML 内容在保存前完成基础清洗：
  - 移除 `script`
  - 移除事件属性
  - 移除危险协议

## 6. 慢网优先前端约束

这是本项目最重要的 UI 和实现边界。

### 6.1 主页面首屏策略

- 首屏优先返回可直接渲染的 HTML 文本，不等待大体积脚本执行。
- 主页面的核心文字区域必须先于编辑增强加载。
- 主页面 CSS 保持极小，优先排版与对比度，不追求复杂视觉特效。
- 主页面脚本按能力分层：
  - 第一层：状态显示、保存、锁定、基础切换
  - 第二层：Markdown 预览增强
  - 第三层：编辑体验增强

### 6.2 降低资源成本的具体策略

- 不使用 Web 字体。
- 不依赖大图、插画、复杂图标库。
- 不用沉重动画。
- 不把后台 JS 打包进前台页。
- 工具栏采用文本按钮或极少量内联 SVG。
- 正文区域优先服务文字可读性，而不是视觉装饰。

### 6.3 页面加载顺序

1. 返回 HTML 与最小内联样式
2. 立即渲染路径、正文、锁定提示、状态位
3. 再挂载保存逻辑
4. 最后加载编辑增强

## 7. UI 方向约束

### 7.1 前台主页面设计目标

- 克制
- 工具感
- 低干扰
- 先文字后功能
- 在超慢网络下仍然能快速看见正文和路径

### 7.2 后台设计目标

- 信息密度可以更高
- 可以更像管理台
- 可接受更丰富的表格、搜索、日志与侧栏

## 8. OpenPencil 多版方向

本轮 UI 比稿只做“方向确认”，不做高保真完整实现。

### 方案 V1：极速单栏

前台特点：

- 顶部极窄状态栏
- 单栏正文区
- 工具栏收敛成一排文本按钮
- 强调最快看到文字

后台特点：

- 左侧窄导航 + 右侧列表/详情
- 偏稳重、偏管理台

适用判断：

- 最符合慢网优先
- 风险最低

### 方案 V2：轻双栏工作台

前台特点：

- 左侧窄信息栏，右侧正文/编辑区
- 锁定状态、路径、保存状态放在侧栏
- 保持克制但更有产品感

后台特点：

- 上方统计卡 + 下方表格
- 右侧抽屉编辑

适用判断：

- 平衡可读性与工具感
- 比 V1 更像成熟产品

### 方案 V3：分栏检视台

前台特点：

- 中央正文区 + 右侧辅助面板
- 更强调“读写工作台”氛围
- 更容易承载预览、锁定、统计

后台特点：

- 多区块卡片化布局
- 更强的系统控制台感

适用判断：

- 视觉辨识度更高
- 但前台首屏复杂度最高，需要严格控重

## 9. 本轮设计输出要求

OpenPencil 比稿必须满足：

- 至少 `3` 个版本
- 每个版本都包含：
  - 前台主页面
  - 后台 Dashboard 概念预览
- 主页面必须突出“慢网先出文字”的设计逻辑
- 页面命名清晰，便于后续删掉比稿只保留正式稿

## 10. 后续执行顺序

1. 先在 OpenPencil 出 `V1 / V2 / V3` 三版方向稿
2. 用户确认方向
3. 删除比稿页，仅保留正式稿
4. 再进入实际项目开发
